## Discussões

### Necessidade de Memória Persistente
Identificamos que Claude Code não mantém contexto entre sessões - cada conversa é completamente independente. Isso cria desafio para continuidade de trabalho em projetos de longo prazo.

### 5 Opções de Implementação Exploradas

1. **Arquivo de Contexto no Repositório**
   - Criar arquivo .claude/memory.md
   - Simples, versionado no git
   - Requer leitura manual no início de sessões

2. **MCP (Model Context Protocol) Server**
   - Servidor externo gerenciando memória
   - Automático, estruturado
   - Mais complexo de configurar

3. **Sistema de Notas no Obsidian**
   - Aproveitar infraestrutura do vault
   - Integrado com Obsidian (links, tags, grafos)
   - Visível na interface do Obsidian

4. **Git Commits + CHANGELOG**
   - Usar histórico git como memória
   - Focado em mudanças técnicas
   - Menos conversacional

5. **Hooks do Claude Code**
   - Scripts automáticos em eventos
   - Requer programação adicional
   - Claude Code nem sempre detecta fim de sessão

### Decisão Final: Combinação 1+3 em .claude/

Inicialmente consideramos combinar opção 1 (arquivo de contexto) com opção 3 (sistema de notas Obsidian), mas ajustamos para manter tudo em `.claude/` por ser conteúdo manipulado apenas via Claude Code.

## Decisões Tomadas

1. **Decisão:** Sistema de memória completo em `.claude/`
   - **Razão:** Mais apropriado para conteúdo que não faz parte do vault principal, usado exclusivamente via Claude Code
   - **Alternativas consideradas:** Pasta `Claude/` visível no Obsidian (descartada por misturar conteúdo de trabalho com metadados)
   - **Impacto:** Mantém vault limpo, separa conteúdo de usuário de conteúdo técnico

2. **Decisão:** Estrutura com múltiplos arquivos especializados
   - **Razão:** Separação de responsabilidades - resumo rápido vs contexto detalhado vs histórico
   - **Componentes:**
     - `memory.md` - Resumo executivo para leitura rápida
     - `contexto-projeto.md` - Contexto profundo do projeto
     - `template-sessao.md` - Template reutilizável
     - `sessoes/` - Histórico de sessões
     - `README.md` - Documentação do sistema
   - **Impacto:** Permite leitura rápida ou profunda conforme necessidade

3. **Decisão:** Workflow manual de leitura
   - **Razão:** Não há automação disponível sem MCP server complexo
   - **Processo:** Usuário pede "Leia .claude/memory.md" no início de cada sessão
   - **Impacto:** Requer disciplina, mas é simples e confiável

## Aprendizados

- Claude Code não mantém memória entre sessões por design
- Sistema de arquivos bem estruturado pode compensar essa limitação
- Separação entre resumo rápido e contexto detalhado oferece flexibilidade
- Templates garantem consistência na documentação
- `.claude/` é local apropriado para metadados de ferramentas
